## 二、数据存储位置详解（数据库在哪）

### 2.1 没有传统数据库

本系统**没有使用**以下任何数据库：
- ❌ MySQL / PostgreSQL / SQL Server
- ❌ MongoDB / Redis
- ❌ 任何云数据库服务

### 2.2 数据实际存储位置：浏览器 localStorage

```
存储位置：浏览器本地 → localStorage → key: "restaurant_data"
数据格式：JSON 字符串
存储容量：约 5MB（浏览器限制）
生命周期：浏览器关闭后数据仍在，清空浏览器缓存后丢失
```

**代码位置**：`js/data.js` 第 2-13 行

```javascript
const STORAGE_KEY = 'restaurant_data';

function getStore() {
  try {
    const d = localStorage.getItem(STORAGE_KEY);  // 从浏览器本地读取
    return d ? JSON.parse(d) : null;            // JSON字符串转对象
  } catch { return null; }
}

function saveStore(data) {
  localStorage.setItem(STORAGE_KEY, JSON.stringify(data));  // 对象转JSON存回浏览器
}
```

### 2.4 数据迁移机制

每次页面加载时，`migrateData()` 函数自动修复旧数据：
- 旧订单状态（cooking/ready）迁移为新状态（pending/served）
- 为旧订单补充 `served`（菜品出餐状态）、`paid`（结账状态）、`tableSessionId`（会话隔离ID）字段
- 从订单数据重新计算菜品销量（确保 sold 字段与订单一致）

**代码位置**：`js/data.js` 第 58-97 行

## 五、核心机制详解

### 5.1 会话隔离机制（解决旧订单残留问题）

**问题**：桌台改为空闲后，新客人选同一桌，仍能看到上一桌的订单。

**解决方案**：每个桌台新增 `currentSessionId` 计数器，每批新客人+1。

```javascript
// 桌台对象
{ id: 1, name: '1号桌', status: 'occupied', currentSessionId: 3 }

// 订单对象
{ id: 'ORD-001', tableId: 1, tableSessionId: 3, items: [...], total: 128 }
```

**数据匹配规则**：订单只显示在 `tableSessionId === currentSessionId` 的桌台上，旧订单自动隔离。

**代码位置**：
- `app.js` 第 137 行：`tbl.currentSessionId = (tbl.currentSessionId || 0) + 1`
- `admin.js` 第 206 行：`o.tableSessionId === table.currentSessionId`

### 5.2 订单号日期制（每日从001开始）

```javascript
function generateOrderNo() {
  const today = new Date().toISOString().slice(0, 10);  // 2026-06-17
  const todayOrders = store.orders.filter(o => o.orderNo?.startsWith(today));
  const maxSeq = todayOrders.reduce((m, o) => {
    const seq = parseInt(o.orderNo?.split('-')[3] || 0);
    return Math.max(m, seq);
  }, 0);
  return `${today}-${String(maxSeq + 1).padStart(3, '0')}`;  // 2026-06-17-001
}
```

### 5.3 菜品出餐状态（逐道菜控制）

每个订单的 `items` 数组中，每道菜都有独立的 `served` 字段：

```javascript
items: [
  { dishId: 1, name: '招牌红烧肉', qty: 2, price: 58, served: true },   // 已出餐
  { dishId: 4, name: '宫保鸡丁', qty: 1, price: 42, served: false }   // 待出餐
]
```

管理员可逐道菜标记出餐，也可一键标记全部出餐。桌台卡片根据所有订单的菜品出餐状态显示整体进度。

### 5.4 结账机制（订单级别）

每个订单有 `paid` 字段（布尔值）：
- 未结账 → 显示「待结账」标签 + 「结账」按钮
- 已结账 → 显示「已结账」绿色标签
- **营业额统计只统计 `paid === true` 的订单**，确保数据真实

## 六、前后端数据交互流程（真实版）

### 6.1 正常情况（有后端时的理想流程）

```
顾客端(app) ──HTTP请求──→ 后端服务器(Node.js/PHP) ──SQL查询──→ 数据库(MySQL)
     ↑                         ↓                           ↓
     └────────JSON响应─────────┘←────────数据返回──────────┘
```

### 6.2 本系统实际流程（纯前端）

```
顾客端(app) ──localStorage读写──→ 浏览器本地存储（同浏览器内）
     ↑                                    ↓
     └────────同一STORAGE_KEY─────────────┘

管理端(admin) ──localStorage读写──→ 浏览器本地存储（同浏览器内）
     ↑                                    ↓
     └────────同一STORAGE_KEY─────────────┘
```

**关键**：顾客端和管理端**必须在同一台电脑的同一个浏览器**中打开，才能共享数据。换电脑或换浏览器后数据不互通。
